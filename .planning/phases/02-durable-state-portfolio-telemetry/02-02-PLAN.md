## Phase Goal

**As** the portfolio owner, **I want** the digest to remember peaks and portfolio value
across runs, **so that** trailing-stops fire on real history and a permanently-down stock
stops generating stale-peak noise. This slice makes the 02-01 peak/corp-action mechanics
durable and adds the daily snapshot + sentiment cache.

<objective>
Turn the three currently-stubbed inputs into real, persisted, bounded state: create
state.py (atomic load/save + snapshot write/prune), add a same-day sentiment cache to
sentiment.py, and rewire sentinel.py to thread real state through rules → sentiment →
snapshot → save. This realizes STATE-01/02/03 end-to-end (peaks now persist so the 02-01
mechanics actually fire) and STATE-04 (date-keyed idempotent bounded snapshots).

Per D-03 the single state.json at repo root holds {peaks, snapshots, sentiment}. Per D-04
new_state is rebuilt each run (rules already does this). Per D-02/STATE-04 snapshots are
keyed per calendar date and a same-day rerun overwrites. Per D-05 snapshots are pruned to
~10 recent entries and sentiment to current holdings. Per D-10 sentiment is cached by
calendar day so hourly reruns cost ~1 model call/day. Atomic write (temp + os.replace) per
02-RESEARCH so a crash never leaves a partial file.

Note (locked): committing state.json back to the repo is Phase 3 (RUN-03) — do NOT add
that here. state.json is a local file this phase; do NOT gitignore it (Phase 3 tracks it).

Purpose: without persistence the whole peak lifecycle and telemetry are inert. Output: a
working state.json round-trip and a cache that collapses N hourly runs into one model call.

- sentiment.adjust new signature `(flags, api_key, cache, today) -> (flags, new_cache)`.
- sentinel.py real state wiring (replaces `state={}`).
</objective>
