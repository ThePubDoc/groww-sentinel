<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → GitHub secrets / healthchecks dashboard | Human provisions the credentials and the external monitor the runtime depends on |
| GitHub Actions runtime → repo | Confirms whether the commit-back token has (or is denied) write access under org policy |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege | commit-back token | high | mitigate | This checkpoint verifies the least-privilege default `contents: write` empirically; only if denied is a fine-grained STATE_PAT (this repo + contents:write only) introduced (D-05). |
| T-03-07 | Repudiation | "green run" that didn't actually send | medium | mitigate | The checkpoint requires confirming the Telegram message AND the healthchecks ping AND the state.json commit — three independent signals, not just a green check (D-07/D-08). |
</threat_model>
