<interface_context>
The workflow runs `python -m sentinel` from the repo root. `sentinel.main()` reads
secrets from `os.environ` — required: `GROWW_API_KEY`, `GROWW_TOTP_SEED`,
`TELEGRAM_TOKEN`, `TELEGRAM_CHAT_ID`; optional: `GEMINI_API_KEY`, `HEALTHCHECK_URL`.
Exit codes: 0 = ok / clean skip, 2 = missing secret, 1 = auth/fetch failure.
`state.py` writes `state.json` at the repo root (atomically) each successful run;
`state.json` is NOT in `.gitignore`, so the commit-back action can add it.
Dependencies are pinned in `requirements.txt` (no new package this phase).
</interface_context>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runtime → repo | The default GITHUB_TOKEN (or STATE_PAT) pushes a commit back — write access to the repo |
| GitHub encrypted secrets → sentinel process env | Four+ secrets cross into the process environment at run time |
| sentinel process → external APIs (Groww/Telegram/healthchecks) | Outbound calls carrying secret material |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege | workflow token / commit-back | high | mitigate | Least-privilege `permissions: contents: write` at workflow scope only, no other scopes; STATE_PAT fallback is fine-grained to this one repo + `contents: write` only, never a classic broad-`repo` PAT (D-05/D-09, 03-RESEARCH Security Domain V4). |
| T-03-02 | Information Disclosure | secrets in `env:` / CI logs | high | mitigate | Secrets sourced from GitHub encrypted secrets, injected via `env:`, never echoed; `sentinel._redact` already strips secret values from any surfaced exception (Phase 1). |
| T-03-03 | Tampering / Repudiation | leaked HEALTHCHECK_URL suppresses the dead-man's-switch | low | accept | Personal single-user monitor; stored as an encrypted secret; worst case is silencing the switch, no financial data or account access exposed (03-RESEARCH Security Domain). |
| T-03-06 | Denial of Service | concurrent/duplicate runs clobbering state.json | medium | mitigate | `concurrency` group + `cancel-in-progress: false` serializes runs (D-06); atomic `state.py` write is the second line of defense. |
</threat_model>
